# HMI 栈统计总览设计

## 1. 目标

在 HMI 中新增更直观的任务栈统计总览能力，支持：

- 实时显示每个任务的栈统计
- 显示总体占用、剩余和风险任务
- 持续将结构化栈统计快照写入本地 `jsonl`
- 保留现有原始日志落盘能力

本次设计以“表格总览”为主视图，不以趋势图为主。

## 2. 背景与现状

当前 HMI 已具备以下基础：

- 端口 B 解析 DGUS 日志帧
- 从日志文本中提取 `STACK_LEVEL=<min>` 单值样本
- 页面 `栈水位统计` 已显示单条趋势图与基础统计
- 原始 HMI 日志会持续写入本地滚动 `jsonl`

当前不足：

- 仅能看到单个最小栈水位，无法区分每个任务
- 无法直接显示“总栈 / 已占用 / 剩余”
- 栈统计没有单独的结构化持久化文件

## 3. 总体方案

采用“固件日志携带完整任务栈快照，HMI 实时解析并结构化落盘”的方案。

方案边界：

- 固件负责输出稳定、可解析的任务栈快照日志
- HMI 负责解析、汇总、展示、写盘
- 不在 HMI 内写死任务总栈配置
- 不新增独立二进制协议，继续复用 DGUS 日志通道

## 4. 固件日志口径

固件需要输出一份“完整快照”而不是零散片段，至少包含 7 个任务：

- `ProtoTask`
- `StateMachineTask`
- `MotorTask`
- `AdcTask`
- `CommTask`
- `MonitorTask`
- `HeaterTask`

每个任务至少提供两个字段：

- 总栈大小
- 当前剩余栈

HMI 将据此计算：

- 当前已占用 = 总栈大小 - 当前剩余栈
- 当前占用率 = 当前已占用 / 总栈大小
- 历史最小剩余
- 历史最大占用

建议固件输出稳定文本格式，要求：

- 同一份快照内字段名固定
- 任务名固定且与 HMI 解析表一致
- 一次快照应可唯一识别开始与结束，或者使用单条完整日志承载全部任务

本设计不锁死具体日志字符串，但要求最终格式满足“单次完整快照可被确定性解析”。

## 5. HMI 数据模型

HMI 新增三类内存态数据：

### 5.1 任务实时统计

每个任务一份当前状态：

- 任务名
- 总栈大小
- 当前剩余栈
- 当前已占用
- 当前占用率
- 历史最小剩余
- 历史最大占用
- 最近更新时间

### 5.2 总览统计

由当前任务状态聚合得出：

- 任务总栈
- 当前总已占用
- 当前总剩余
- 最危险任务

“最危险任务”默认定义为“当前剩余最小”的任务；若剩余相同，则取占用率更高者。

### 5.3 结构化快照流

每次收到完整快照后，生成一份结构化快照对象：

- 时间戳
- 总览统计
- 全部任务统计列表

该对象既用于 UI 刷新，也用于单独落盘。

### 5.4 长期运行内存约束

为避免 HMI 长时间运行时内存持续增长，实现必须满足：

- 原始日志内存缓存继续保留上限
- 栈趋势样本只保留有限窗口
- 结构化栈快照只保留有限窗口
- 半截快照拼装缓存必须有上限，超限后自动丢弃并复位
- 历史数据依赖 `jsonl` 落盘，不常驻内存

## 6. 页面设计

保留现有菜单项“栈水位统计”，但页面内容改为以下结构：

### 6.1 顶部总览区

显示 4 张卡片：

- 总栈
- 当前总已占用
- 当前总剩余
- 最危险任务

### 6.2 主表格区

每个任务一行，列建议为：

- 任务名
- 总栈
- 当前剩余
- 当前已占用
- 占用率
- 历史最小剩余
- 历史最大占用
- 更新时间

直观性要求：

- 高占用率或低剩余任务使用明显高亮
- 表格在窄屏下仍需避免溢出
- 数值列使用等宽字体

### 6.3 辅助区

原有单值趋势图不作为主视图。

首版处理建议：

- 可以保留为辅助模块，显示最近一次“全局最小剩余”趋势
- 也可以先隐藏，只保留表格与总览

为控制范围，首版优先完成“总览 + 表格 + 持久化”。

### 6.4 页面操作

保留“清空样本”操作，但语义调整为：

- 只清空 HMI 当前内存中的栈统计样本与聚合态
- 不删除已经落盘的原始日志和结构化栈统计文件

## 7. 持久化设计

持久化分为两层：

### 7.1 原始日志

继续沿用现有滚动 `jsonl` 文件，不做兼容性破坏。

### 7.2 栈统计结构化日志

新增单独滚动 `jsonl` 文件，按“每次完整快照一行”持续追加。

每行建议包含：

- `ts`
- `summary`
- `tasks`

其中：

- `summary` 保存总览统计
- `tasks` 保存全部任务的当前结构化状态

这样可以支持后续：

- 离线分析
- 回放
- 导出
- 增加趋势图

## 8. 解析与兼容策略

HMI 解析策略应满足以下要求：

- 只有识别到完整快照时才更新表格与结构化日志
- 不因半截日志或噪声日志污染当前任务统计
- 若部分任务字段缺失，则本次快照丢弃并记录调试日志
- 若遇到未知任务名，则忽略该任务并标记兼容性问题

首版不要求兼容多套旧格式；以当前固件联调目标格式为准。

## 9. 实现范围

### 9.1 固件侧

- 调整栈统计日志输出格式
- 确保总栈与剩余栈可稳定输出
- 确保单次快照具备完整性

### 9.2 HMI 控制器

- 新增任务栈快照解析逻辑
- 新增任务实时统计与总览聚合
- 新增结构化栈统计 `jsonl` 落盘
- 提供页面读取与清空接口

### 9.3 HMI 页面

- 重构“栈水位统计”页主内容
- 增加总览卡片
- 增加任务表格
- 调整清空按钮语义

## 10. 非目标

本次不包含以下内容：

- 新增独立串口协议功能码
- 让 HMI 手工维护任务总栈配置
- 首版就实现每任务独立趋势图
- 删除现有原始日志滚动文件

## 11. 测试与验证

至少验证以下场景：

- 正常接收完整快照后，表格与总览正确刷新
- 原始日志继续正常落盘
- 新结构化栈统计 `jsonl` 正常持续追加
- 清空内存统计后，页面归零但磁盘文件保留
- 遇到残缺快照或异常日志时，不污染当前统计
- 窄屏布局无 `RenderFlex overflow`
- 长时间连续接收快照时，内存态样本与快照数量不会无限增长

## 12. 风险与注意事项

- 固件日志格式若不稳定，HMI 解析会脆弱，因此必须先定格式再实现
- 若单次快照被拆成多条 DGUS 文本，必须设计明确的边界标识
- 若日志频率过高，结构化 `jsonl` 体积会增长较快，需要复用现有滚动写盘策略

## 13. 推荐实施顺序

1. 先确定固件栈快照日志格式
2. 在 HMI 控制器实现解析与结构化落盘
3. 重构栈统计页面为“总览 + 表格”
4. 补充异常格式与窄屏验证
